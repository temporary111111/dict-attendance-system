Treat this as a real internal government-office system that needs to be practical, maintainable, secure, and privacy-conscious.

System being planned:
The system is attendance-related, but it is NOT an employee daily time-in/time-out system. It is for managing attendance of external attendees/clients for DICT programs and events.

Best current system name/framing:
**Program and Event Attendance Monitoring and Reporting System for DICT**

More specific version:
**QR-Based Program and Event Attendance Management System with Google Forms Integration**

Goal:
I want help planning this properly before coding. I want the system to be professional and realistic, not overengineered. Focus first on solid scope, MVP, requirements, database design, user roles, workflow, and roadmap.

Supervisor’s minimum requirements/scratch notes:

Features:

Other specific supervisor notes:

* External attendees do not need accounts.
* Admin creates an event under a program.
* System should generate a QR code and/or link for the attendance form.
* External users open the QR/link and input their details.
* For names, use standard format: separate first name, middle name, last name, and suffix.
* Use PSGC Code for address.

Important update:
The attendance form is currently Google Forms. So the system should not immediately be treated as a custom form builder unless we explicitly decide to make that Phase 2.

Current best understanding:
Google Forms is the attendance collection layer. The system should be the admin/event/reporting layer around it.

Correct MVP framing:
The system manages programs/events, stores Google Form links, generates QR codes, imports or syncs responses, validates attendance data, generates reports, enforces RBAC, and records audit logs.

Do NOT overpromise that the app itself generates the full attendance form unless Google Forms API/Apps Script or a custom built-in form is actually implemented.

Current workflow:

1. Admin logs in.
2. Admin creates/manages programs.
3. Admin creates events under programs.
4. Each event has a Google Form attendance link.
5. System stores the event’s Google Form link.
6. System generates a QR code pointing to the Google Form link, ideally a pre-filled Google Form link with an event code.
7. External attendee scans QR/opens link.
8. External attendee submits attendance through Google Forms.
9. Google Forms responses are exported or synced.
10. System imports/reads the responses.
11. System validates and stores attendance records in MySQL.
12. Admin views dashboard and reports per event/program.
13. System records audit trail for important admin actions.

Important decision already made:
Same-person detection across events is useful, but it should NOT be part of the MVP. It is too advanced for now and can cause complexity in identity matching, duplicate detection, privacy, and database design.

For MVP, focus only on duplicate detection within the same event.

Reasoning:
Core first:

1. Reliable event attendance capture
2. Clean import process from Google Forms/Sheets
3. Clean reports
4. RBAC and audit trail
5. Stable program/event management
6. Then advanced identity matching later

For MVP duplicate detection:

* Same event + same email = duplicate or possible duplicate
* Same event + same mobile number = duplicate or possible duplicate
* Name-only matching is weak and should not be relied on

RBAC logic:

* Super Admin can manage all programs, events, users, reports, imports, exports, and audit logs.
* Program Admin can only manage assigned programs and events under those assigned programs.
* Program Admin should not access programs/events outside their assignment.
* External Attendee has no account and only submits attendance through Google Forms.

Important real-office requirements:

* Audit trail for important actions
