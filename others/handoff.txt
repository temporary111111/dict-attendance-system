Context:
Intern/OJT ako ngayon sa Department of Information and Communications Technology (DICT). Computer Science ang course ko. May pinapagawa sa amin na system ng DICT supervisor namin, and possible itong gamitin sa actual office workflow. Hindi ito thesis-only project. Treat this as a real internal government-office system that needs to be practical, maintainable, secure, and privacy-conscious.

System being planned:
The system is attendance-related, but it is NOT an employee daily time-in/time-out system. It is for managing attendance of external attendees/clients for DICT programs and events.

Best current system name/framing:
Program and Event Attendance Monitoring and Reporting System for DICT

More specific version:
QR-Based Program and Event Attendance Management System with Template-Based Attendance Sheet Generation

Goal:
Plan this properly before coding. The system should be professional and realistic, not overengineered. Focus first on solid scope, MVP, requirements, database design, user roles, workflow, and roadmap.

Supervisor's minimum requirements/scratch notes:

Features:

* Login for admin
* Attendance UI for external user
* Dashboard summary
* Programs / adding of programs
* Adding of event per program
* Reports generation per event/program
* Audit trail
* Implement RBAC

Roles:

* Super Admin
* Admin per program
* External clients / external attendees

Database:

* MySQL
* Implement normalization, at least 2NF or 3NF

Other specific supervisor notes:

* External attendees do not need accounts.
* Admin creates an event under a program.
* System should generate a QR code and/or link for the attendance page.
* External users open the QR/link and input their details.
* For names, use standard format: separate first name, middle name, last name, and suffix.
* Supervisor provided a DICT attendance sheet PDF template. This should be followed for the downloadable attendance sheet/report generated by the system.

Important update:

The system is no longer connected to Google Forms.

The system should also NOT become a dynamic form builder. The attendee input should be fixed. The template provided by the supervisor is the report/output format that the admin downloads once the event is done.

Current best understanding:

The system should collect attendance directly through a fixed public event attendance page, store the records in MySQL, and generate a downloadable DICT attendance sheet using the supervisor-provided template.

Correct MVP framing:

The system manages programs/events, generates event attendance QR codes and public links, collects fixed attendance submissions, validates attendance data, generates template-based attendance sheets, generates reports, enforces RBAC, and records audit logs.

Do NOT overpromise:

* Google Forms integration
* Google Sheets sync
* CSV import as the main attendance flow
* Dynamic form builder
* Biometrics/facial recognition
* Cross-event identity matching

Current workflow:

1. Admin logs in.
2. Admin creates/manages programs.
3. Admin creates events under programs.
4. Admin opens attendance collection for the event.
5. System generates a public attendance link and QR code.
6. External attendee scans QR/opens link.
7. External attendee submits attendance through the fixed system page.
8. System validates and stores the attendance record in MySQL.
9. Admin views attendance records and dashboard data.
10. Admin closes the event after attendance collection.
11. Admin downloads the event attendance sheet using the DICT template.
12. System records audit trail for important actions.

Important decision already made:

Same-person detection across events is useful, but it should NOT be part of the MVP. It is too advanced for now and can cause complexity in identity matching, duplicate detection, privacy, and database design.

For MVP, focus only on duplicate detection within the same event.

For MVP duplicate detection:

* Same event + same email = duplicate or possible duplicate
* Name-only matching is weak and should not be relied on

RBAC logic:

* Super Admin can manage all programs, events, users, reports, downloads, and audit logs.
* Program Admin can only manage assigned programs and events under those assigned programs.
* Program Admin should not access programs/events outside their assignment.
* External Attendee has no account and only submits attendance through the public event attendance page.

Important real-office requirements:

* Password hashing
* Strict RBAC enforcement
* Audit trail for important actions
* Soft delete/archive instead of hard delete where appropriate
* Report/download export logging
* Search/filter/pagination
* Privacy notice and consent handling
* Avoid collecting unnecessary personal data
* Do not collect government ID unless explicitly required
* Do not add biometrics/facial recognition
* Documentation should be clear enough for future DICT staff/admins to maintain
